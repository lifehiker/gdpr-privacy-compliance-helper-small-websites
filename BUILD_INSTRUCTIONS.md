# GDPR & Privacy Compliance Helper for Small Websites

*Generated by Forge on 2026-04-17T06:47:04.390Z — score 8.2/10*

## Overview
A lightweight web SaaS for indie developers, freelancers, agencies, and small website owners to run a fast privacy compliance audit, see what is missing, and track fixes over time. The problem is not that small sites need enterprise-grade legal software; it is that they need a practical way to check basics like consent banners, policy links, tracking scripts, forms, data flows, retention notes, and audit evidence without paying for a full compliance suite. The solution is a self-serve “privacy audit + fix tracker” app that combines automated website scanning with a guided checklist and evidence capture, positioned clearly for small websites rather than as a generic GDPR platform.

## Target user
Primary persona: a solo founder, freelance web developer, or small agency owner managing 1–20 websites.

Who they are:
- They run marketing sites, SaaS landing pages, ecommerce stores, or brochure websites.
- They use common tools like Google Analytics, Meta Pixel, Stripe, Mailchimp, HubSpot, Typeform, Calendly, Intercom, or contact forms.
- They are not lawyers and do not want to become compliance experts.

What they currently do:
- Manually check whether a site has a privacy policy, cookie banner, consent settings, and form disclosures.
- Use spreadsheets, Notion docs, or ad hoc task lists to track compliance work.
- Sometimes rely on cookie banner vendors or policy generators, but those tools do not give them a practical audit workflow across all privacy basics.
- For client work, they take screenshots and save loose notes in Google Drive or Slack.

Why current options fail them:
- Enterprise tools are too expensive and too broad.
- Cookie-banner-first tools solve only one slice of the problem.
- Legal doc generators do not help them identify missing implementation details on the website.
- Existing compliance suites often feel heavy, legalistic, or overbuilt for a small operator who just wants to know: what is missing, how serious is it, and what should I fix first?

## Core features
- User authentication with Google sign-in and email-based account support later if needed.
- Website/project management:
  - Create a project with site name, domain, region focus (EU/global), and notes.
  - View all projects in a simple dashboard.
- Automated homepage/site scan:
  - Fetch and analyze the homepage HTML.
  - Detect presence of privacy policy link, cookie policy link, terms link, contact form, newsletter/email signup form, analytics scripts, common marketing pixels, chat widgets, embedded third-party scripts, and payment provider references.
  - Store scan timestamp and findings.
- Rule-based privacy checks:
  - Generate a checklist status from scan results:
    - Privacy policy present/missing
    - Cookie/consent banner likely present/missing
    - Analytics detected
    - Marketing trackers detected
    - Forms collecting personal data detected
    - Payment/data processor indicators detected
    - Third-party embeds/scripts detected
  - Mark each item as Pass / Warning / Fail / Needs manual review.
- Manual audit checklist:
  - Editable checklist for non-detectable items:
    - Data retention documented
    - Data processor list documented
    - Legal basis documented
    - DSAR/contact process documented
    - Consent logging process defined
    - Cookie categories documented
  - User can mark status, add notes, and assign due dates.
- Remediation tracker:
  - Convert failed/warning items into tasks.
  - Task fields: title, severity, status, owner, due date, notes.
  - Kanban is not needed; simple list view only.
- Evidence capture:
  - Upload screenshots/files for each project or checklist item.
  - Add URL evidence and text notes.
  - Store uploaded files in S3-compatible storage.
- Audit history:
  - Keep scan history per project.
  - Show “last scan vs current scan” summary.
- Basic compliance score:
  - Simple weighted score out of 100 based on checklist completion and scan findings.
  - Show top 3 missing items to fix first.
- PDF export:
  - Export a simple audit summary PDF with project info, findings, checklist status, tasks, and evidence references.
- Billing/subscriptions:
  - Free tier with one project and limited scans.
  - Paid plan for more projects, recurring scans, exports, and evidence storage.
- Email notifications:
  - Send welcome email.
  - Send weekly digest for overdue tasks and scan reminders.
- Marketing site:
  - Landing page, pricing page, feature page, blog, and free website privacy check lead capture page.

## Tech stack recommendation
Use the standard web stack. This is a browser-based B2B SaaS with dashboards, scans, storage, recurring subscriptions, and SEO-driven acquisition. It does not require native mobile capabilities. A web app ships faster, supports search acquisition, and matches how users manage websites.

Recommended stack:
- Next.js 15.x (App Router) + TypeScript 5.x
- PostgreSQL 16 + Prisma ORM 6.x
- Tailwind CSS 3.4+ + shadcn/ui
- NextAuth v5 (`next-auth@beta` / Auth.js v5) with Google OAuth
- Stripe for subscriptions and customer portal
- Resend for transactional email
- Coolify deployment via Docker
- S3-compatible object storage for evidence uploads:
  - Prefer Cloudflare R2 or Backblaze B2
- Background job processing:
  - Use database-backed scheduled jobs initially via Vercel/cron-equivalent is not ideal on Coolify, so implement app-level scheduled route plus server cron in Coolify hitting internal endpoints
- PDF generation:
  - `@react-pdf/renderer` or HTML-to-PDF via Playwright
- HTML parsing and scanning:
  - `cheerio`
- HTTP fetching:
  - native `fetch` with timeout wrapper, optionally `axios` if preferred
- File upload:
  - `uploadthing` is optional, but for simplicity use presigned S3 upload flow
- Validation/forms:
  - `zod`, `react-hook-form`
- Tables:
  - `@tanstack/react-table`
- Date handling:
  - `date-fns`
- Charts:
  - `recharts` for simple score/history chart
- Logging/error tracking:
  - `pino` or `console` initially; optional Sentry later, not MVP
- Optional crawler helpers:
  - `robots-parser` to respect robots.txt for deeper scans if added later

Why this stack:
- SEO matters, so Next.js is the right delivery model.
- PostgreSQL + Prisma keeps relational project/checklist/task/evidence data easy to manage.
- Stripe and NextAuth are standard and fast to integrate.
- Coolify is cost-effective for a solo founder.
- The scanner can be implemented server-side without needing browser extension or native app complexity.

## Zero-Config Deployment Requirement

**CRITICAL: This app MUST start and run correctly with NO environment variables configured in the deployment platform (Coolify). All external service dependencies must either be absent or have safe fallback defaults baked in.**

Apps that crash on startup due to missing env vars will fail every deployment cycle.

**CRITICAL: Never instantiate third-party API clients at module level.** Next.js evaluates top-level code during `next build` — a `new Resend(...)`, `new Stripe(...)`, or `new OpenAI(...)` call at module scope will crash the build if the API key env var is missing. Always initialize these clients **inside** the handler/function body, guarded by an env var check.

## Key Implementation Notes

### Database (only if PRD requires persistence)
- Use **SQLite** via Prisma: `provider = "sqlite"`, `url = env("DATABASE_URL")`
- `DATABASE_URL` defaults to `file:/data/app.db` in the Dockerfile — no Coolify config needed
- For local dev: `DATABASE_URL="file:./dev.db"` in `.env.local`
- Run `npx prisma db push` after schema changes
- **Do NOT use PostgreSQL** — no PostgreSQL server is provisioned in the deployment environment

### Authentication (only if PRD requires user login)
- Use **NextAuth v5** with the **Credentials provider** (email + password)
- **Do NOT use Google OAuth** — no OAuth credentials are configured in deployment
- Hash passwords with **bcrypt** (`bcryptjs` package)
- Session strategy: **`"jwt"`** (not `"database"`) — required for Credentials provider
- `AUTH_SECRET` defaults are baked into the Dockerfile; app works out of the box

### Payments (only if PRD explicitly requires it)
- Use Stripe only when the PRD's core functionality requires payment processing
- If adding Stripe: handle `checkout.session.completed`, `customer.subscription.updated`, `customer.subscription.deleted` at `/api/webhooks/stripe`
- Make the app functional without Stripe credentials (show upgrade prompts rather than crashing)

### Email (only if PRD explicitly requires it)
- Use Resend only when the PRD requires transactional email
- **CRITICAL: NEVER instantiate `new Resend(...)` at module level** — always inside the handler function body with a guard:
  ```ts
  // WRONG — causes build failure (next build evaluates module-level code):
  const resend = new Resend(process.env.RESEND_API_KEY);
  export async function POST(req) { ... }

  // CORRECT — lazy init inside handler:
  export async function POST(req) {
    if (!process.env.RESEND_API_KEY) {
      console.log("[email] RESEND_API_KEY not set, skipping email");
      return NextResponse.json({ ok: true });
    }
    const { Resend } = await import("resend");
    const resend = new Resend(process.env.RESEND_API_KEY);
    ...
  }
  ```
- Next.js evaluates module-level code during `next build` page data collection — missing env vars at module scope crash the build
- Same rule applies to Stripe, OpenAI, and ALL third-party SDK clients — always lazy-initialize inside handler functions

### Fonts
- **Do NOT use `next/font/google`** — it fetches font files during `next build`, which can hang or fail in automated/offline build environments.
- Use CSS/system font stacks by default.
- If a custom font is truly required, commit the font file into the repo and use `next/font/local`; never depend on a build-time network font fetch.

### Deployment
- Coolify with Docker — always set `output: "standalone"` in next.config.ts
- Do NOT add `COPY public ./public` in the Dockerfile if the public/ directory is empty or doesn't exist
- The Dockerfile's CMD initializes the SQLite database on first start via `prisma db push`
- App must respond healthy on port 3000 within 60 seconds of container start
- **CRITICAL Dockerfile rule**: In the `deps` stage, always use `RUN npm ci --ignore-scripts`. Without `--ignore-scripts`, the `postinstall` hook (e.g. `prisma generate`) runs before `prisma/schema.prisma` is copied, causing the build to fail. Run `npx prisma generate` explicitly in the `builder` stage after `COPY . .`.
- **Do NOT run `npx prisma db push` in the Dockerfile builder stage** — it is not needed at build time. The CMD already runs `prisma db push` at container startup when a real `/data/app.db` path is available.
- **Use `node:20-slim` (not `node:20-alpine`)** — Debian slim avoids Alpine musl/libssl issues. Add `RUN apt-get update -y && apt-get install -y openssl && rm -rf /var/lib/apt/lists/*` to BOTH the builder and runner stages for Prisma's schema engine.
- **Set `binaryTargets` in schema.prisma** — always include `binaryTargets = ["native", "debian-openssl-3.0.x"]` in the Prisma generator block so Prisma generates the correct engine binary for the Debian container environment.
- **Copy full node_modules in the runner stage** — do NOT selectively copy `node_modules/.prisma`, `node_modules/prisma`, `node_modules/@prisma` separately. Instead use `COPY --from=builder --chown=nextjs:nodejs /app/node_modules ./node_modules` to copy everything. Prisma v6+ CLI has deep transitive deps (`@prisma/config` → `effect`, `c12`, etc.) that break if not all present.
